# Agent Project Roadmap

本文将当前项目中已经识别出的功能缺口整理成一份 roadmap，重点不是做一个通用的大而全 agent 平台，而是围绕“教学型 agent 项目”这个目标，优先补齐最能提升演示价值、理解价值和可扩展性的能力。

## 当前项目的定位

从现在的实现来看，这个项目已经是一个结构清楚、适合讲解 agent 基础概念的 MVP。

已经具备的核心能力包括：

- 多轮 chat loop
- 本地工具调用
- tool hallucination guard
- Markdown RAG
- CLI 交互
- JSONL debug log

这意味着它已经适合教学这些基础主题：

- 什么是 agent loop
- 工具调用是怎么工作的
- RAG 是怎么接入 prompt 的
- 模型为什么会产生 fake tool call
- 如何做最基本的 agent 调试

但如果目标是“做成一个完整的教学项目”，当前还缺少几类关键能力。

## Roadmap 设计原则

本 roadmap 按以下原则排序：

1. 优先补齐最能提升教学演示效果的功能。
2. 优先做能明显增强 agent 闭环能力的功能。
3. 在核心演示体验完整之前，不优先做企业级扩展。
4. 区分用户可见功能与工程硬化项，避免路线图混在一起。

## 总体阶段划分

建议把后续演进分成 3 个阶段：

- Phase 1: 教学核心能力补齐
- Phase 2: 产品完整性与可用性增强
- Phase 3: 稳定性、扩展性和规模化支持

## Phase 1: 教学核心能力补齐

这一阶段的目标是：让项目从“能讲基础原理”升级为“能完整演示 agent 如何解决问题”。

### 1. 受控 Python 执行工具

#### 当前状态

这个能力已经实现，当前内置工具现在包含：

- `get_current_time`
- `add_numbers`
- `run_python_code`
- `list_workspace`
- `read_workspace_file`

这些工具适合展示最基础的 function calling，也已经可以支撑更完整的 agent 演示。

#### 为什么重要

在教学场景里，Python 执行几乎是最有价值的工具类型，因为它能直观展示：

- 模型不会的事情可以交给工具
- 工具运行结果如何反馈给模型
- 模型如何基于运行结果修正答案

#### 建议能力范围

- 提供一个受控的 Python REPL 或单次执行工具
- 限制可用模块或运行目录
- 限制执行时间和输出长度
- 将 stdout、stderr 和异常结构化返回给模型

#### 建议优先级

最高优先级。

---

### 2. Streaming 输出

#### 当前缺口

当前 CLI 需要等完整回复结束后才显示结果。

#### 为什么重要

对于教学项目来说，streaming 比“最终答案是否漂亮”更重要，因为它能帮助观察：

- 模型是直接回答还是先思考工具调用
- 模型何时停顿
- 工具调用前后的输出变化
- 空回复或异常回复出现在哪个阶段

#### 建议能力范围

- 支持流式打印 assistant 内容
- 在 tool call 前后显示状态变化
- 区分普通文本流和工具事件

#### 建议优先级

最高优先级。

#### 为什么 streaming 特别适合教学演示

streaming 之所以适合教学，不只是因为“看起来更实时”，而是因为它能把 agent 的内部过程暴露得更清楚。对于教学项目来说，这一点非常关键。

第一，streaming 能把“模型是在想，还是在调用工具”这个过程直接展示出来。非流式输出里，用户只看到最终答案，很难判断模型是在什么节点开始组织语言、什么时候准备调用工具、什么时候因为上下文变化而重新改口。流式输出会让这些变化更明显。

第二，streaming 很适合讲解 tool calling。很多初学者会把 agent 理解成“模型直接回答问题”，但实际上 agent 的核心是“先生成一段内容，再根据需要触发工具，再把工具结果喂回模型”。如果能够一边输出一边展示，就更容易让人看到：模型内容和工具事件不是一次性完成的，而是分阶段发生的。

第三，streaming 便于调试和演示失败路径。比如：

- 模型突然停住
- 先输出一半又转去调用工具
- 空回复后触发重试
- 检索到 RAG 但模型没有直接用上

这些情况在教学里都很有价值，因为它们能帮助解释“agent 并不总是稳定地一次给出完美答案”。流式输出会让这些边界案例更加可见。

第四，streaming 能让课堂演示更自然。对于讲解型项目来说，现场展示时等待完整答案往往会削弱节奏，而流式输出会让观众更容易跟着你一起观察模型的行为，而不是只看结果。

所以，streaming 在这个项目里不是一个“锦上添花”的 UI 特性，而是一个帮助理解 agent 工作方式的核心教学能力。它能同时提升：

- 可观察性
- 讲解性
- 调试性
- 演示节奏

### 3. 会话导出功能

#### 当前缺口

虽然已经有 JSONL debug log，但缺少“把一次教学演示导出成可读文档”的能力。

#### 为什么重要

教学项目通常需要：

- 回放一次演示过程
- 把课堂或 demo 结果发给别人
- 对比同一个问题在不同配置下的行为

#### 建议能力范围

- 导出 Markdown 会话记录
- 导出结构化 JSON 会话记录
- 可选包含：用户输入、工具调用、RAG 命中、最终回答

#### 建议优先级

高优先级。

---

### 4. 更强的教学场景模板

#### 当前缺口

当前项目有 demo knowledge base，但缺少成套的演示脚本和 scenario 模板。

#### 为什么重要

教学项目不只是“能运行”，更重要的是“容易讲”。

#### 建议能力范围

- 预置 demo prompts
- 预置故障演示场景
- 预置 RAG on/off 对比场景
- 预置 tool hallucination guard 演示场景

#### 建议优先级

高优先级。

---

## Phase 2: 产品完整性与可用性增强

这一阶段的目标是：让项目从“可演示”变成“可持续使用”。

### 5. RAG 索引管理增强

#### 当前缺口

当前只支持：

- 单文件 `rag add`
- `rag search`
- `rag clear`

#### 为什么重要

这足够展示最小 RAG 流程，但对真实使用和教学扩展来说还不够。

#### 建议能力范围

- 支持目录级批量导入
- 支持重建索引
- 支持查看索引统计信息
- 支持删除单个 source 文件对应的 chunk
- 支持列出当前已索引文档

#### 建议优先级

高优先级。

---

### 6. 更丰富的 RAG 数据源

#### 当前缺口

现在只支持 Markdown。

#### 为什么重要

Markdown 很适合教学起步，但用户很快会问：

- 能不能读 txt
- 能不能读 PDF
- 能不能读 HTML

#### 建议能力范围

- 增加纯文本支持
- 增加 HTML 支持
- 后续再考虑 PDF

#### 建议优先级

中高优先级。

---

### 7. 对话上下文管理

#### 当前缺口

当前多轮对话是不断累积 messages，没有 token budgeting、历史裁剪或摘要压缩。

#### 为什么重要

教学项目特别适合演示“上下文越来越长时 agent 为什么会变差”。如果没有对应机制，项目本身也很容易在长会话里退化。

#### 建议能力范围

- 设置最大上下文预算
- 超限时裁剪旧消息
- 可选生成历史摘要
- 对 system、tool、rag context 分别设置保留策略

#### 建议优先级

高优先级。

---

### 8. 更丰富的工具能力

#### 当前缺口

工具能力仍然太轻量。

#### 为什么重要

如果要把项目作为 agent 教学平台，工具生态必须再扩展一些，至少能覆盖：

- 本地计算
- 文件写入
- 外部信息获取

#### 建议能力范围

- 写文件工具
- 受控 shell 工具
- HTTP / Web API 工具
- 更丰富的数学或数据分析工具

#### 建议优先级

中高优先级。

---

### 9. 自定义工具注册机制

#### 当前缺口

工具注册当前偏硬编码，不利于教学扩展。

#### 为什么重要

教学项目很适合逐步添加工具。如果每加一个工具都要改核心代码，会降低可扩展性，也不利于教学中解释“agent 框架层”和“工具层”的区别。

#### 建议能力范围

- 支持从模块加载工具
- 支持自定义 tool registry 配置
- 支持 demo 模式下只启用部分工具

#### 建议优先级

中优先级。

---

## Phase 3: 稳定性、扩展性和规模化支持

这一阶段的目标是：让项目不仅适合教学，也具备更好的工程质量和扩展空间。

### 10. 更强的错误恢复与重试机制

#### 当前缺口

现在已经处理了：

- fake tool call
- unknown tool call
- empty response

但还缺少更完整的故障恢复。

#### 建议能力范围

- Ollama 请求超时重试
- 临时连接失败恢复
- embedding 请求失败恢复
- 索引损坏检测与修复建议

#### 建议优先级

中优先级。

---

### 11. 更完整的可观测性

#### 当前缺口

有 debug log，但缺少统计和结构化指标。

#### 为什么重要

教学中如果能展示这些指标，会更容易解释 agent 行为：

- 每轮请求耗时
- 工具调用次数
- RAG 命中数量
- 空回复重试次数
- token 使用量

#### 建议能力范围

- 记录每轮耗时
- 记录 tool / rag / retry 统计
- 展示简单会话 summary

#### 建议优先级

中优先级。

---

### 12. 更强的 RAG 检索质量

#### 当前缺口

当前检索方式是最基础的 embedding + cosine similarity。

#### 为什么重要

对教学 MVP 来说已经足够，但如果后续文档规模变大，就会出现：

- 相似 chunk 过多
- 引用过宽泛
- 检索相关性不稳定

#### 建议能力范围

- reranking
- metadata filter
- hybrid retrieval
- chunk 策略可配置

#### 建议优先级

中优先级。

---

### 13. 多模型策略

#### 当前缺口

当前基本是单模型聊天，加一个 embedding 模型。

#### 为什么重要

教学中经常会遇到这样的问题：

- 一个模型适合工具调用
- 一个模型适合回答
- 一个模型更稳定但更慢

#### 建议能力范围

- chat model / tool model 分离
- fallback model
- 按任务路由模型

#### 建议优先级

中低优先级。

---

### 14. 更完整的测试与文档体系

#### 当前缺口

当前测试已经覆盖了不少关键点，但仍然缺少：

- CLI 端到端测试
- config / env 测试
- 错误恢复测试
- 长对话行为测试

#### 为什么重要

教学项目也需要稳定。如果 demo 时行为不稳定，教学价值会明显下降。

#### 建议能力范围

- 补齐 e2e tests
- 补齐 config tests
- 补齐 failure-mode tests
- 增加教程型文档矩阵

#### 建议优先级

中优先级。

---

## 推荐的实现顺序

如果要避免路线图过大、过散，建议按下面顺序推进：

### 第一批

1. 受控 Python 执行工具
2. Streaming 输出
3. 会话导出

这一批完成后，项目会从“基础 agent 样例”升级成“很适合现场讲解的 agent 教学项目”。

### 第二批

1. RAG 索引管理增强
2. 对话上下文管理
3. 工具扩展能力

这一批完成后，项目会从“演示用”升级成“可以持续用来做实验和教学”的平台。

### 第三批

1. 更完整的错误恢复
2. 更强的可观测性
3. RAG 质量增强
4. 多模型策略

这一批完成后，项目会更接近一个成熟的本地 agent 教学框架。

## 最终建议

如果只选 3 个最值得马上做的功能，建议优先实现：

1. 受控 Python 执行工具
2. Streaming 输出
3. RAG 索引管理增强

原因很简单：

- Python 执行最能体现 agent 的价值
- Streaming 最能提升教学体验
- RAG 管理增强最能提升项目的持续可用性

这 3 项组合起来，能用最小复杂度换来最大的教学收益。
